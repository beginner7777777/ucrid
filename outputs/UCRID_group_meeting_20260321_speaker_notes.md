## 第 1 页：标题页

这一页我主要想介绍的是我的研究主题和汇报范围。我的工作聚焦在意图识别和 OOS 检测任务上，核心问题是如何把一个轻量小模型和大语言模型结合起来，形成一个既高效又准确的级联系统。整个方法我称为 UCRID，也就是 uncertainty-aware cascade routing。

这次汇报我会从四个部分展开。第一部分是研究动机，也就是为什么不能只用小模型，或者只用大模型。第二部分是程序和代码逻辑，我会结合实际实现介绍训练、路由和 LLM 仲裁三部分。第三部分是目前在 CLINC150 和 Banking77 上的实验结果。最后一部分是当前结论和后续准备投稿时还需要补的工作。

---

## 第 2 页：研究动机与问题定义

这一页主要回答一个问题，就是为什么要做级联，而不是直接选一种模型。

如果只用小模型，比如 BERT-base，优势是速度快、部署成本低，但它的 OOS 检测能力通常不足，尤其在边界样本和语义模糊样本上，容易把 OOS 误判成某个已知意图。也就是说，它适合处理大部分简单样本，但不适合处理不确定区域。

如果只用大模型，准确率通常更高，尤其是 OOS 识别能力会更强，但是代价是推理延迟和调用成本都明显上升。如果对所有请求都调用 LLM，这个系统在线上就会非常重。

所以我的目标不是在小模型和大模型之间二选一，而是让小模型处理大多数容易样本，只把少量不确定样本交给 LLM 进一步仲裁。这样就形成了一个效率和性能折中的级联框架。

---

## 第 3 页：方法总览

这一页是整个方法的总体框架图，可以把它理解成训练阶段和推理阶段共同构成的完整系统。

先看左半部分，也就是离线阶段。首先，我们用训练数据训练一个 BERT 小模型分类器。接着，从训练集表示中构造每个 in-domain intent 的原型向量。然后，在验证集上做温度缩放和双阈值搜索，得到路由器的参数。与此同时，还要从训练样本中构造 Stage 3 所需要的意图描述、few-shot 示例池和 OOS 示例池。

再看右半部分，也就是在线推理。输入一个 query 之后，先通过 Stage 1 得到 logits 和 embedding。然后在 Stage 2 里计算不确定性分数，并依据双阈值决定这个样本是直接接受小模型、直接判为 OOS，还是送到 Stage 3 给 LLM 精判。最后再根据接纳策略得到最终预测。

这一页建议大家把握一个核心点：UCRID 不是“LLM 直接替代分类器”，而是“LLM 只处理不确定区域”的一个层次化决策系统。

## 第 4 页：训练阶段逻辑

这一页我介绍 Stage 1 的训练逻辑。程序入口在 `src/train.py`，损失函数定义在 `src/losses/multi_task_loss.py`。

从模型结构上看，Stage 1 使用的是一个 BERT-base 编码器加两层 MLP 分类头。模型输出两类结果，一类是分类 logits，另一类是 CLS 表示向量。这个表示向量后面不仅用于分类，也用于构造原型和计算样本到原型的距离。

训练损失不是普通交叉熵，而是三部分联合：交叉熵、监督对比损失和边界损失。交叉熵保证基本分类能力；SupCon 负责把同类样本拉近、异类样本推远；Boundary loss 则专门针对 OOS 样本，让它们远离所有 ID 原型，从而在表示空间中形成更清晰的 OOS 区域。

另外，这个训练过程不是三种损失从头一起开，而是用了 phased schedule。前几轮先只用 CE，让模型先学会基本分类；中间再加 SupCon 去优化嵌入空间；最后再加 Boundary，把 OOS 边界显式建出来。这个设计在实现上非常重要，因为它能避免训练初期过多约束导致模型不稳定。

## 第 5 页：推理阶段逻辑

这一页重点讲推理流程，也就是 `run_ucrid.py` 和 `ucrid_router.py` 里的核心逻辑。

对于一个输入 query，Stage 1 会输出 logits 和 embedding。Stage 2 不是直接用 max softmax probability，而是融合两个信号：一个是温度校准后的 softmax 熵，另一个是该样本到最近 intent 原型的欧氏距离。熵表示分类不确定性，距离表示这个样本是否远离所有已知意图簇。

这两个信号经过归一化后按权重 α 融合，得到不确定性分数 s(u)。接着用双阈值进行路由。如果 s(u) 很低，就直接接受小模型 top-1；如果 s(u) 很高，同时又离所有原型都很远，就直接判成 OOS；只有中间那部分不确定带，才会真正送去 Stage 3。

所以从系统设计来看，Stage 2 是整个方法的关键。它决定了哪些样本应该快速放行，哪些样本可以直接拒绝，哪些样本值得花更高成本交给 LLM。

---

## 第 6 页：Stage 3 的 LLM 仲裁模块

这一页讲 Stage 3。这里的核心思想是，LLM 不是全量分类器，而是一个不确定样本的仲裁器。

送进 LLM 的 prompt 不是只有原始 query，而是一个受约束的 prompt。里面包含四类信息：第一是小模型给出的 top-k 候选意图；第二是每个候选意图的描述；第三是从训练集里动态选出来的 few-shot in-domain 示例；第四是 few-shot 的 OOS 示例。这样 LLM 的决策空间就被限制在一个相对小而明确的候选集合中。

LLM 输出以后，也不是直接当成最终结果，而是先经过解析。程序里会优先尝试读取 JSON label，如果失败再做规则匹配和相似度兜底。之后还要经过接纳策略，比如 `oos_only`，意思是只有当 LLM 明确输出 OOS 的时候，才允许覆盖 Stage 2 的小模型结果。

这一页我建议强调一个现实经验：当前实验中，性能最好的并不是参数最大的模型，而是输出更稳定、更容易被解析、更容易被 `oos_only` 接纳的模型。

## 第 7 页：代码结构与数据流

这一页主要帮助组里同学从工程角度理解代码是怎么组织的。

训练相关的主文件有三个。`src/train.py` 负责 Stage 1 训练；`src/models/bert_encoder.py` 定义了 BERT 和分类头；`src/losses/multi_task_loss.py` 定义了 CE、SupCon 和 Boundary 三种损失。

推理相关的核心文件也有三个。`src/experiments/run_ucrid.py` 是完整实验入口，它负责加载模型、构建原型、做阈值搜索并运行测试；`src/inference/ucrid_router.py` 实现 Stage 2 的路由器；`src/inference/llm_judge.py` 实现 Stage 3 的 prompt 构造、LLM 调用和输出解析。

结果输出也分两类。一个是 `ucrid_results.json`，用于保存 Accuracy、OOS F1、延迟、LLM 调用率等汇总指标；另一个是 `ucrid_prediction_details.json`，用于保存每个样本的路由决策、top-k、uncertainty score、llm_raw 等细粒度信息。这个样本级文件对论文里的错误分析和 case study 很重要。

## 第 8 页：实验结果 CLINC150

这一页展示 CLINC150 上不同开源模型的结果对比。

首先看总体结论。当前在 CLINC150 上效果最好的还是 Qwen3-8B 的 fixed 版本，它在 Accuracy、OOS recall 和 OOS F1 上都明显优于另外两组。尤其是 OOS F1 达到了 0.8431，是目前这套流程下的最佳结果。

为什么 Qwen3-8B fixed 会更好？从我们保存的细粒度结果来看，一个重要原因是它在 `oos_only` 策略下真正被接纳的样本更多，也就是 `llm_used` 更高。这说明它不仅会输出 OOS，而且输出更稳定、更容易被解析和接纳。

相比之下，Mixtral 虽然也比 Stage 2 有提升，但效果还是落后于 Qwen3，同时延迟显著更高。它的一个现实问题是必须使用更短的上下文配置才能在当前 3090 环境里稳定跑通，这会压缩 prompt 信息量，最终影响效果。

所以 CLINC150 这组实验��明：在这个数据集上，Stage 3 的关键不是“模型越大越强”，而是“输出越稳越能被系统正确吸收”。

## 第 9 页：实验结果 Banking77

这一页是 Banking77 的结果。这里和 CLINC150 有一个明显不同点，就是 Qwen2-7B 基线和 Qwen3-8B fixed 之间的差距变小了。

这说明 Banking77 相对来说边界更规整，数据集本身对 Stage 3 的格式稳定性和语义细粒度区分的要求没有 CLINC150 那么极端。所以一个较强的开源 7B 模型也能拿到很不错的结果。

不过从综合指标看，Qwen3-8B fixed 仍然是最稳的方案。Mixtral 这里的问题依旧不是 precision，而是 recall。它能保持很高的 OOS precision，但很多该翻转成 OOS 的样本没有真正被接纳，所以最终 OOS F1 还是落后。

同时延迟差距也非常明显。Qwen2 和 Qwen3 的 P95 基本都在几十毫秒量级，而 Mixtral 已经接近秒级。所以如果同时考虑性能和工程可用性，当前 Banking77 上最有竞争力的方案仍然是 Qwen3-8B fixed。

## 第 10 页：当前结论与下一步计划

最后这一页是总结。

首先，从方法层面，我认为 UCRID 的核心贡献在于用不确定性路由把小模型和 LLM 结合起来，而不是把 LLM 当成一个全量替代方案。这个系统的关键优势是：大多数简单样本仍然由小模型低���本处理，只有不确定区域才调用 LLM，从而在性能和效率之间找到平衡。

其次，从实验上看，当前两个数据集上最优的综合配置都是 Qwen3-8B fixed。它不是最大的模型，但在输出稳定性、可解析性、以及 `oos_only` 策略下的可接纳性方面表现最好。

最后，从投稿准备的角度看，下一步还有三件事值得继续补。一是做更完整的消融实验，例如去掉 SupCon、Boundary、distance signal 或 Stage 3；二是补成本分析，把 LLM 调用率和延迟讲清楚；三是用 prediction details 做错误案例分析和可解释性分析。把这三部分补上之后，论文说服力会更完整。
